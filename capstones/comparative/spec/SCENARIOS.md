# Conformance scenarios and fixture runner contract

This document defines how language-local runners execute the JSON fixtures for
`comparative-kv` version `1.0.0`.

## 1. General runner requirements

The runner receives a language-local `PROGRAM` launcher as a list of executable
and fixed arguments. It appends each fixture `args` array exactly, without shell
evaluation. It MUST NOT build one shell command string.

For every CLI invocation the runner captures raw stdout, raw stderr, exit code,
and elapsed time. Unless a fixture uses a specialized assertion, `expect`
means:

1. exit code equals `exit`;
2. stderr bytes decode as UTF-8 and equal `stderr`;
3. stdout is UTF-8 with no BOM;
4. stdout contains exactly one compact JSON object followed by LF;
5. when `stdout` is present, the parsed object is semantically equal to it; and
6. every emitted JSON number uses the normalized rules in `SPEC.md`.

When `stdout` is omitted, a structural assertion in the same operation MUST
fully inspect the parsed envelope. Omitting both `stdout` and a structural
assertion is invalid fixture data.

Semantic equality compares object member sets and values without considering
object member order or string escape spelling. It compares array order,
strings by exact Unicode scalar sequence, booleans, null, and safe integers.
No test normalizes Unicode.

Each scenario receives a new local directory and `${DB}` expands to a database
path inside it. `${MISSING_PARENT}` expands to a path whose parent does not
exist. Substitutions are individual argument values, never shell text.

The runner validates every fixture's `spec_version` against `SPEC_VERSION`
before execution. Unknown fixture keys or generator kinds fail the runner
rather than being ignored.

## 2. Common fixture kinds

### 2.1 `constants`

`contract.json` is a machine-readable index of frozen limits, grammar, and exit
codes. A runner SHOULD assert its local constants agree with it.

### 2.2 `key_cases`

For each accepted case in `keys.json`, run set/get in a fresh database and
require success. For each rejected case, run `get` and require:

```json
{"ok":false,"error":{"category":"invalid_argument","details":{"field":"key","reason":"format"}}}
```

A `key_generator` of `repeat_suffix` creates `prefix + character * count`.
The generated result, not the descriptor, is the command argument.

The `ordering` array is inserted in reverse order with unconditional set, then
`list` MUST return keys in exactly the listed order. Revisions attached to
those entries reflect insertion order and are not reordered with the keys.

### 2.3 `accepted_value_cases`

For each case, use a fresh database and:

1. run `set value --value-json INPUT --expect absent`;
2. require exit 0, `created:true`, revision 1, and `value` equal to
   `normalized`;
3. run `get value`; and
4. require the same normalized value and revision 1.

`input_json` is passed as the exact JSON-text argument.

Supported `input_generator` values:

- `nested_arrays`: wrap `leaf` in one-element arrays `depth` times and serialize
  compactly;
- `ascii_string_total_bytes`: create one JSON string containing repeated
  `character` so the complete compact JSON text, including quotes, has exactly
  `total_bytes` UTF-8 bytes.

`normalized_generator` constructs the corresponding semantic value using the
same parameters. Generators MUST first assert the generated byte length or
container depth claimed by the descriptor.

### 2.4 `rejected_value_cases`

For each case, use a fresh database and run:

```text
--db ${DB} set value --value-json INPUT
```

Require the case's exit/category/details, then assert `${DB}` was not created.
Generators have the meanings from section 2.3.

## 3. Sequential scenario manifest

Files with kind `sequential_scenarios` contain isolated scenarios. `database`
is:

- `fresh`: `${DB}` does not exist before the first step; or
- `sqlite_setup`: execute `setup.statements` in order using a separate SQLite
  connection, close it, then run CLI steps.

Each `run` step appends its `args` to `PROGRAM` and uses the standard `expect`
algorithm. All steps run even after an expected command failure. An unexpected
failure aborts the scenario but still performs cleanup.

The setup connection MUST use SQLite itself, not implementation-private code.
SQL strings are fixture data and are executed one statement at a time without
text substitution.

A `sqlite_assert` step opens the database through an independent SQLite
connection after the preceding CLI process has exited. Each listed query is
read-only and its complete result rows, in query order, MUST equal `rows`.
SQLite integers, text, null, and blob byte strings retain their distinct
types. The assertion connection is closed before the next step. This operation
is used only to verify schema rollback or state that cannot be observed after
an intentionally rejected migration; it never replaces CLI assertions.

`boundary.json` may contain a `fixture_references` step. It invokes the complete
named fixture files using sections 2.2-2.4 at that point.

After every scenario, run SQLite integrity checking when a database exists,
unless the expected result is corruption or malformed storage that cannot be
opened safely.

## 4. Parallel scenario operations

Files with kind `multiprocess_scenarios` use these operations.

### 4.1 `parallel`

Start all actors as independent child processes. Use a start barrier so every
child has been created before any child is released to execute `PROGRAM`.
Capture each result independently and apply the listed assertions after all
children exit.

`actors_generator.indexed_commands` expands `count` actors. `${i}` is the
zero-based integer and `${n}` is `i + 1` in every template string. Generated
keys use the decimal `${n}` padded to `pad_width` when `${padded_n}` appears.

Supported aggregate assertions:

- `all_exit`: every exit code equals the value;
- `all_ok`: every parsed envelope has that boolean `ok`;
- `stdout_semantic_all`: every parsed stdout equals the object;
- `success_count`, `category_counts`: exact result partition;
- `result_revision_set`: sorted successful `result.revision` values equal the
  inclusive integer range;
- `success_revision`: every successful mutation has that revision;
- `conflict_actual`: every conflict has that `details.actual`;
- `not_found_count`: exact count of `not_found`;
- `winner_value_matches_final`: final stored value equals the value argument of
  the sole successful actor; and
- `duration_less_than_ms` / `duration_at_least_ms`: coarse hang/busy bounds.

### 4.2 `run_assert`

Run one CLI process, then apply both the ordinary `expect` object and optional
structural assertions:

- `keys_in_order`;
- `global_revision`;
- `entry_count`;
- `entry_revision_set`;
- `values_by_key`;
- `revision_by_key`; and
- `state_unchanged_from` a named captured result.

### 4.3 Captures

`capture` stores a parsed result under an identifier. A later assertion may
refer to it. Captures never expose implementation-private state.

### 4.4 SQLite lock helper

`start_lock_helper` launches a separate OS process, not a thread in the test
runner. The helper:

1. opens `${DB}`;
2. configures a busy timeout;
3. executes `BEGIN IMMEDIATE`;
4. signals `ready` to the runner only after acquiring the lock; and
5. waits for the runner's `release` signal before rollback and close.

`start_cli` launches a CLI child without waiting. `sleep_ms` is allowed only to
ensure a lock is held for a minimum interval after the helper's ready signal;
it is never used to predict mutation winner order. `await_cli` waits for and
asserts the named child. `release_lock_helper` signals, waits for, and verifies
clean helper exit.

For the timeout case, the helper remains locked until after the CLI returns
`busy`; the runner then releases it. The CLI process timeout is at least
15,000 ms. The expected elapsed lower bound is 9,500 ms to allow timer
granularity while still proving that a materially shorter timeout was not
configured.

### 4.5 SQLite setup in process scenarios

`sqlite_setup` has the same meaning as in section 3. It completes and closes
before the start barrier. For migration-race scenarios it creates exact v0 and
MUST NOT run the implementation before the actors start.

## 5. Nondeterminism rules

The fixtures intentionally do not predict:

- which distinct writer receives which revision;
- which same-key CAS actor wins;
- which delete actor acquires the lock first;
- child-process completion order; or
- object member serialization order.

They do require exact counts, revision sets, final state, and error details.
Runners MUST NOT weaken an aggregate assertion by accepting additional
outcomes.

## 6. Repetition and timeouts

`repeat` means rerun the whole scenario with a new directory and database.
The minimum process timeout is:

- 15 seconds for a single ordinary CLI;
- 30 seconds for a parallel group;
- 20 seconds for a busy-timeout CLI; and
- 30 seconds for cleanup after failure.

A repository MAY use longer outer CI timeouts. It MUST NOT skip a required
repeat or turn a timing failure into a retry that hides the failed attempt.

## 7. Cleanup assertions

Cleanup follows `SPEC.md` section 10 after every repeat. Before deleting the
scenario directory, the runner asserts that it can remove `${DB}`,
`${DB}-wal`, `${DB}-shm`, and `${DB}-journal` when present. A child that
survives its timeout is terminated by its recorded process identifier and
waited for; name-based process killing is forbidden.

## 8. Required fixture coverage

- `normal.json`: ordinary set/get/list/delete, all expectations, semantic
  object behavior, revision gaps, and delete/recreate.
- `boundary.json`: accepted/rejected keys and values, exact limits, ordering,
  same-value mutation, and validation precedence.
- `invalid.json`: grammar, path forms, keys, expectations, JSON syntax, and
  error envelopes.
- `migration.json`: exact v0, deterministic assigned revisions, rollback on
  invalid legacy data, future/malformed schemas, and revision exhaustion.
- `multiprocess.json`: fresh initialization, distinct writers, CAS race,
  failed-CAS accounting, delete race, migration race, busy release, busy
  timeout, and cleanup.

Passing only sequential fixtures is not version-1 conformance.
